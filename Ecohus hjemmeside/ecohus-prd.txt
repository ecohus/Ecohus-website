Architecture Rules
Must Use:
	•	Server Components by default — Client Components only where interactive (forms, calculator, lightbox)
	•	Zod schemas for all form validation — both client-side and server-side API route validation
	•	Next.js Image component for all images — WebP format, sizes prop required on every instance
	•	ISR (Incremental Static Regeneration) for Sanity-driven pages — never SSR for static content
	•	Environment variables for all API keys — nothing hardcoded

Must Avoid:
	•	Inline styles — Tailwind utility classes only
	•	any TypeScript types — strict mode enabled
	•	Client-side data fetching for page-level content — use generateStaticParams + ISR
	•	"use client" on layout or page components — only on leaf interactive components

4. FEATURES

Feature 1: Forside (Homepage)   [P0 — MVP]

User Story
As a potential summer house buyer, I need to immediately understand what Ecohus offers, feel confident in their quality, and have a clear next step so I can decide whether to explore further or book a consultation.

Acceptance Criteria
	•	Hero section renders with full-bleed house photo, H1 headline ("Færdigbyggede huse direkte til din grund" or updated copy), primary CTA "Book en samtale" and secondary CTA "Se vores modeller"
	•	"5 gode grunde" value props section renders with 5 points matching current ecohus.dk content (fixed price, fast delivery, eco-friendly, factory-built quality, permit assistance)
	•	Process teaser section shows 4 horizontal steps: Produktion → Transport → Opsætning → Klar til brug with icons and short labels
	•	Featured models section renders 3 house model cards (Sanity CMS, is_featured: true). Each card: main image, model name, m², price from (kr.), "Se model" CTA
	•	Testimonials section: renders empty state ("Anmeldelser tilføjes snart") when Sanity testimonials array is empty; renders testimonial cards with name, quote, star rating when populated
	•	"Er du klar?" CTA banner renders at bottom with "Book en samtale" button
	•	All sections fully responsive: 375px (mobile), 768px (tablet), 1440px (desktop)
	•	Hero image loads with priority={true} — LCP target met < 2.5s

UI/UX Notes
	•	Hero: 100vh desktop, 70vh mobile. Text left-aligned on desktop, centered on mobile. Photo covers full width with subtle dark overlay gradient (20% opacity) for text legibility
	•	Value props: 2-column grid desktop, 1-column mobile. Green checkmark icon per item. Light green (#EAF2EC) background for this section
	•	Process teaser: 4 boxes in a horizontal row with thin green connecting line between them. On mobile: vertical stack with left border accent line
	•	Featured models: 3-column grid desktop, 1-column mobile. Cards have white background, subtle border, hover state (scale + shadow)
	•	Reference: [REF: frifeldt-hero.png] for hero layout · [REF: huscompagniet-hero.png] for CTA pattern

Technical
	•	Server Component — no client state needed on this page
	•	ISR revalidation: 60 seconds for featured models query from Sanity
	•	Hero image: stored in Supabase Storage or Sanity asset, optimized via next/image
	•	Testimonials: GROQ query from Sanity, ordered by date desc, limit 6

Feature 2: Plantegninger (House Models Page)   [P0 — MVP]

User Story
As a potential buyer, I need to browse all available house models, filter them by size and features, and click through to a detailed page for any model that interests me.

Acceptance Criteria
	•	Page renders grid of all active house models from Sanity CMS (is_active: true), ordered by size_m2 ascending
	•	Filter controls: size range slider (m²), room count (1–6+), garage toggle (all / yes / no)
	•	Filtering updates model grid without page reload (client-side JS filter, data loaded once)
	•	URL search params update on filter change (?size=60-120&rooms=3&garage=true) — URLs are shareable
	•	Each model card displays: main image, model name, size in m², number of rooms, price from (kr.), "Se model" CTA button
	•	Empty state shown ("Ingen modeller matcher dit filter — prøv andre kriterier") if filters return 0 results
	•	Page renders correctly with 0 models from CMS (graceful empty state: "Modeller tilføjes snart")
	•	Reference: [REF: frifeldt-plantegninger-grid.png]

Technical
	•	"use client" component for filter state and URL param updates
	•	Sanity GROQ: *[_type == "house_model" && is_active == true] | order(size_m2 asc)
	•	ISR revalidation: 120 seconds

Feature 3: Plantegning Detail (/plantegninger/[slug])   [P0 — MVP]

User Story
As a buyer exploring a specific model, I need to see the floor plan, measurements, full spec sheet, and included features, then be able to book a consultation about this specific model.

Acceptance Criteria
	•	Page generates statically for all active model slugs via generateStaticParams()
	•	Returns 404 for non-existent or inactive model slugs
	•	Displays: model name, m², rooms, bathrooms, price from, main image, floor plan PNG
	•	Floor plan PNG is zoomable — click opens fullscreen lightbox (Radix Dialog). Lightbox has close button and click-outside-to-close
	•	Specs table: size (m²), rooms, bathrooms, has garage (Ja/Nej), has canopy (Ja/Nej), construction type
	•	"Hvad er inkluderet" features list: rendered from Sanity features[] array
	•	3D model embed: renders iframe if embed_3d_url field is populated in Sanity; hidden if empty
	•	"Book en samtale om denne model" CTA links to /kontakt?model=[model-name] (pre-fills model field in contact form)
	•	"Se lignende modeller" section at bottom: 3 models within ±20m² of current model
	•	Reference: [REF: frifeldt-model-detail.png]

UI/UX Notes
	•	Desktop layout: 2-column. Left (60%): main image + floor plan. Right (40%): name, price, specs, features, CTA
	•	Mobile layout: single column. CTA button is sticky fixed at bottom of viewport
	•	Floor plan: max-width 600px, object-fit: contain, white background. Click to fullscreen.

Technical
	•	Static generation: generateStaticParams() fetches all active slugs at build time
	•	ISR revalidation: 300 seconds
	•	Lightbox: @radix-ui/react-dialog (already in shadcn/ui) — "use client" component

Feature 4: Prisberegner (Price Calculator)   [P0 — MVP]

User Story
As a potential buyer, I need to get an estimated price for my summer house configuration so I can evaluate whether Ecohus fits my budget before committing to a consultation call.

Acceptance Criteria
	•	Multi-step calculator with 4 steps and visible step progress indicator
	•	Step 1 — Vælg model: Grid of model cards (from Sanity). User selects one. Shows model name, m², price from. "Selected" state visible (green border + checkmark)
	•	Step 2 — Tilvalg: Checkbox options — garage (+ kr. 120.000), overdækning/carport (+ kr. 45.000), additional insulation (+ kr. 35.000). Live price total updates as options are toggled
	•	Step 3 — Fundament: Radio select — skruefundament (+ kr. 85.000), støbt betonfundament (+ kr. 110.000). Short explanation text per option
	•	Step 4 — Din pris: Displays line-item breakdown and total range (e.g. "kr. 1.450.000 – kr. 1.620.000"). Note: "Prisen er vejledende — inkl. levering og opsætning, ekskl. tilslutninger og tilladelser"
	•	Lead capture form on step 4: Navn (required), E-mail (required), Telefon (required), Kommentar (optional)
	•	Back button available on all steps (step 1 back button is disabled/hidden)
	•	On form submit: lead saved to price_calculator_leads Supabase table + email notification to kontakt@ecohus.dk via Resend
	•	Confirmation screen: "Tak! Vi vender tilbage inden for 1–2 hverdage." with "Tilbage til forsiden" link
	•	Form validation: client-side + server-side (Zod). Field-level error messages in Danish
	•	Reference: [REF: jvrconsult-prisberegner.png] for step UI and price display style

UI/UX Notes
	•	Centered card layout, max-width 760px, white card on warm off-white page background
	•	Progress indicator: 4 pill-shaped step indicators at top (active = green fill, complete = green outline + checkmark, upcoming = gray)
	•	Step transitions: slide animation (framer-motion) — slide left when advancing, slide right when going back
	•	Price display on step 4: large green typography, bold total, line items in smaller text below
	•	Mobile: full-width card, "Næste →" button fixed at bottom of viewport

Price Logic
// Base prices come from Sanity model.price_from field
// Option prices are hardcoded in config (can be moved to Sanity later):
const OPTION_PRICES = {
  garage:     120000,
  canopy:      45000,
  insulation:  35000,
};
const FOUNDATION_PRICES = {
  screwpile:   85000,
  concrete:   110000,
};
// Total range: base_price_from + options + foundation = min
// max = min + 12% (contingency buffer — displayed as range)

Technical
	•	"use client" component — multi-step state managed with useState
	•	POST /api/calculator-lead: Zod validation → Supabase insert → Resend email
	•	Rate limited: max 5 submissions per IP per hour (Upstash Redis)

5. DATA & INTEGRATIONS

Database Schema (Supabase)
-- Lead capture from contact form
CREATE TABLE contact_submissions (
  id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name        VARCHAR(255) NOT NULL,
  email       VARCHAR(255) NOT NULL,
  phone       VARCHAR(50)  NOT NULL,
  message     TEXT,
  model_interest VARCHAR(255),
  source      VARCHAR(100) DEFAULT 'contact_form',
  created_at  TIMESTAMPTZ  DEFAULT NOW()
);

-- Lead capture from price calculator
CREATE TABLE price_calculator_leads (
  id                  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name                VARCHAR(255) NOT NULL,
  email               VARCHAR(255) NOT NULL,
  phone               VARCHAR(50)  NOT NULL,
  model_selected      VARCHAR(255),
  options             JSONB,
  estimated_price_from INTEGER,
  estimated_price_to   INTEGER,
  created_at          TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX idx_contact_submissions_created ON contact_submissions(created_at DESC);
CREATE INDEX idx_calculator_leads_created ON price_calculator_leads(created_at DESC);

Sanity CMS Schema
house_model
name, slug, size_m2 (number), rooms (number), bathrooms (number), has_garage (bool), has_canopy (bool), price_from (number), main_image (image), floorplan_png (image), description (text), features (string[]), is_active (bool), is_featured (bool), embed_3d_url (string optional), order (number)
gallery_image
image (image), alt (string), category (eksteriør|interiør|byggefase), order (number), is_active (bool)
testimonial
name (string), quote (text), rating (1-5), date (date), is_active (bool)
team_member
name, title, photo (image), bio (text), order (number)
faq_item
question (string), answer (text), order (number)
site_settings
company_name, phone, email, address, stat_1_label, stat_1_value, stat_2_label, stat_2_value, stat_3_label, stat_3_value, meta_title, meta_description

API Endpoints
POST /api/contact
Submit contact form. Auth: none (public). Zod validate → Supabase insert → Resend notify. Rate: 5/hour/IP. Returns: 200 {success} | 400 {error: validation} | 500 {error: generic}
POST /api/calculator-lead
Submit calculator lead. Auth: none (public). Zod validate → Supabase insert → Resend notify. Rate: 5/hour/IP. Returns: 200 {success} | 400 {error} | 500 {error}

Third-Party Integrations
Sanity.io
Headless CMS. SDK: @sanity/client. Project ID: set at setup. Dataset: production. Auth: SANITY_API_TOKEN (read-only token for frontend). Sanity Studio deployed at studio.ecohus.dk or sanity.io/manage.
Resend
Transactional email. SDK: resend. From: noreply@ecohus.dk. Notify: kontakt@ecohus.dk. API key: RESEND_API_KEY. Domain must be verified in Resend dashboard.
GA4
Google Analytics 4. Fresh account setup needed. Implemented via next/script (gtag). Track: page_views, form_submit (contact + calculator), model_view events. No PII in event parameters.
Upstash Redis
Rate limiting. SDK: @upstash/ratelimit + @upstash/redis. Used on /api/contact and /api/calculator-lead. Config: 5 requests per hour per IP using sliding window.

6. USER FLOWS

Flow 1: Lead Conversion via Model Detail
Trigger
User lands on homepage
Step 1
Reads hero, clicks "Se vores modeller" → /plantegninger
Step 2
Browses model grid, applies filters (optional)
Step 3
Clicks model card → /plantegninger/[slug]
Step 4
Reviews floor plan, specs, features, price from
Step 5
Clicks "Book en samtale om denne model" → /kontakt?model=[name]
Step 6
Contact form pre-filled with model name. Completes Navn, E-mail, Telefon, Besked.
Step 7
Submits form → inline confirmation
Success
contact_submissions row in Supabase. Notification email to kontakt@ecohus.dk.
Edge: Sanity down
Model pages fall back to last ISR cache. User still reaches contact form.

Flow 2: Price Calculator Lead
Trigger
User clicks "Beregn din pris" CTA (homepage or nav)
Step 1
Calculator loads at Step 1. User selects base model.
Step 2
Selects options (garage, canopy, insulation). Price updates live.
Step 3
Selects foundation type. Price updates.
Step 4
Views price breakdown. Fills in Navn, E-mail, Telefon.
Step 5
Submits → confirmation screen
Success
price_calculator_leads row in Supabase. Email notification sent.
Edge: missing fields
Zod validation shows Danish field-level error messages. Form not submitted.
Edge: API error
Generic Danish error shown. User can retry or use phone number.

11. ARCHITECTURE DECISIONS

ADR 1: Sanity.io as CMS (vs custom Supabase admin)
Decision
Use Sanity.io headless CMS for all client-editable content
Reason
Client explicitly requires self-editing capability for gallery, house models, FAQ, and team. Building a custom Supabase-backed admin panel would add significant dev time. Sanity has a free tier sufficient for this use case and an excellent Next.js integration.
Alternatives considered
Custom Supabase admin panel (more dev time, more maintenance), Contentful (more expensive), WordPress headless (heavier stack)
Trade-offs accepted
Adds Sanity subscription if content exceeds free tier limits. Sanity Studio is a separate URL. Antigravity team must define Sanity schemas at setup.

ADR 2: Supabase for lead storage (vs email-only)
Decision
Store all form submissions (contact + calculator leads) in Supabase in addition to sending email notifications
Reason
Email-only provides no audit trail. Client will want to review leads historically, export to a future CRM, and have a backup if email is missed. Supabase free tier is sufficient.
Alternatives considered
Email-only (simpler, no DB setup for leads), direct CRM integration (no CRM identified yet)
Trade-offs accepted
Slightly more setup. Supabase requires a separate lead review interface (Supabase dashboard is sufficient for now).

ADR 3: ISR over SSR for Sanity-driven pages
Decision
Use Incremental Static Regeneration (ISR) for all Sanity-driven pages rather than Server-Side Rendering (SSR)
Reason
Content (house models, gallery, FAQ) changes infrequently. ISR gives near-static performance with automatic content refresh. SSR would add latency on every request and unnecessary Sanity API calls.
Alternatives considered
Full SSR (fresh on every request), full static (no auto-refresh without redeploy)
Trade-offs accepted
Sanity edits take up to the revalidation window (60–300s) to appear on live site. Client informed of this behaviour.

ADR 4: Claude API not used on this project
Decision
No AI/LLM integration in Site 1 (Ecohus Sommerhus)
Reason
Site is a marketing and lead generation platform. No feature requirement for AI content generation or chat. Adding Claude API would add cost and complexity without user value.
Alternatives considered
AI chatbot for lead qualification (potential Phase 2 feature if client requests)
Trade-offs accepted
None. Can be added in a future phase.
